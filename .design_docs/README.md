# Design Docs 目录规范

所有设计文档都应放在此目录下。

## 文档命名规范

- 使用 `snake_case` 命名
- 后缀表示文档类型：
  - `*_design.md` - 功能设计文档
  - `*_plan.md` - 实现计划文档
  - `*_guidelines.md` - 规范指南文档

**注意：** YAML 配置文件应放在 `latest_resumes/` 目录下，而不是此目录

## 文档结构

### 设计文档 (*_design.md)

```markdown
# 功能名称设计

## 概述
简要说明功能目标和背景

## 现状分析
分析当前状态和存在的问题

## 设计方案
详细说明设计方案和实现思路

## 关键决策
记录重要的设计决策和理由

## 预期成果
说明实现后的预期效果
```

### 实现计划 (*_plan.md)

```markdown
# 功能名称实现计划

## 目标
明确实现目标

## 实施方案
分阶段说明实现步骤

## 文件修改清单
列出需要修改的文件

## 时间估计
估计各阶段所需时间

## 风险与缓解
识别风险并提出缓解方案
```

### 规范指南 (*_guidelines.md)

```markdown
# 规范名称指南

## 概述
说明规范的目的和适用范围

## 规范内容
详细说明规范要求

## 示例
提供具体示例

## 常见问题
回答常见问题
```

## 最佳实践

1. **版本控制** - 设计文档应纳入 git 版本控制
2. **链接** - 在文档中链接相关代码文件（使用 `file_path:line_number` 格式）
3. **更新** - 实现完成后更新相关设计文档
4. **审查** - 重要设计文档应在 PR 中审查
5. **归档** - 过时的设计文档可移到 `.snapshots/` 保存
